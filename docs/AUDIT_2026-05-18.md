- Auth（注册、登录、短信 OTP、社交 OAuth 框架）

### 3.3 实时性说明

位置追踪走 HTTP 轮询。一条 10+ 人的远征是可用的，但快速变位会丢点。Socket.io 目前**只**用于聊天，没有用于 location。位置追踪已升级为 Socket.io 实时推送，HTTP POST 作为降级兜底（PR-36）。

## 4. 数据库 Schema（Prisma）

### 4.1 schema 与路由的不一致

Prisma `schema.prisma` 与原始 SQL `db/database.js` 是**两套并行、部分分歧**的系统：

- [x] **`Booking` 模型**（`schema.prisma:628`）缺失字段：`club_id`、`club_name`、`type`、`confirmed_at`、`rejected_reason`、`pool/is_pool`。（PR-35 已补齐）
      raw `database.js:310` 通过 `ALTER TABLE` 补 `club_id`，运行时可用但 Prisma 不同步。
- [x] **`platform_transactions` 与 `withdrawal_requests` 表**：被 `routes/pay.js` 重度使用（托管、结算、提现），但 **Prisma schema 完全没建模**。仅存在于 `database.js:2271-2305`。（PR-35 已纳入 schema）
- [x] **`stripe_webhook_events` 表**：`routes/payment.js:210` 使用，仅在 `database.js:2456` 定义。（PR-35 已纳入 schema）
- [ ] **`user_locations` 表**：`routes/location.js:24-37` 动态 `CREATE TABLE IF NOT EXISTS`——Prisma 完全没有。
- [x] **`Guide` 模型**（schema:285）缺字段：`listing_fee_paid` 和 `listing_fee_paid_at`。路由 `routes/guides.js:397-411` 用 raw SQL `UPDATE guides SET listing_fee_paid = 1` 读写。（PR-35 已补齐）
- [ ] **新商户模型**（`Merchant`、`PlatformExpedition`、`ExpeditionSku`、`PlatformOrder`、`Payout`、`InviteCode`，schema 1345-1560）已完整定义，但 **没有任何路由引用**——v2 架构尚未落地，是死代码。
- [x] **`sos_alerts.status` 列**：`routes/sos.js:91,121` 使用，但不在 `SosAlert` Prisma 模型内；运行时靠 `ALTER TABLE ADD COLUMN` 兜底。（PR-35 已补齐）
- [x] **`Notification` 模型**（schema:1195）缺字段：`title`、`body`、`link`。路由 `notifications.js:13` 直接 select 这些列。（PR-35 已补齐）

### 5.2 PII_ENCRYPTION_KEY 关键问题

`utils/crypto.js:16-27`：
```js
if (!KEY_HEX) {
  if (process.env.NODE_ENV === 'production') {
    throw new Error('FATAL: PII_ENCRYPTION_KEY is not set in production');
  }
  // dev：使用硬编码 'summitlink_dev_pii_key_32bytes!!'
}
```

**风险**：所有在 dev/staging 环境注册、用 dev key 加密手机号/邮箱的用户，一旦生产环境设置真实 `PII_ENCRYPTION_KEY` 后，密文将无法解密。需要明确的 dev→prod 数据迁移方案。
